Flex弹性布局

发表于 2018-03-05 | 分类于 CSS

Flex基本概念

  • Flex 是 Flexible Box 的缩写，意为”弹性布局”，用来为盒状模型提供最大的灵活性。任何一个容器都可以指定为 Flex 布局。
  • Webkit 内核的浏览器，必须加上-webkit前缀。
  • 设为 Flex 布局以后，子元素的float、clear和vertical-align属性将失效。

父容器的属性

flex-direction

决定主轴的方向，即项目的排列方向

  • row（默认值）：主轴为水平方向，起点在左端。
  • row-reverse：主轴为水平方向，起点在右端。
  • column：主轴为垂直方向，起点在上沿。
  • column-reverse：主轴为垂直方向，起点在下沿。

flex-wrap

决定项目在轴线上排不下的时候的换行行为

  • nowrap（默认）：不换行。
  • wrap：换行，第一行在上方。
  • wrap-reverse：换行，第一行在下方。

flex-flow

是flex-direction 和 flex-wrap属性的简写形式，默认值：row nowrap

justify-content

定义项目在主轴上的对齐方式

  • flex-start（默认值）：左对齐
  • flex-end：右对齐
  • center： 居中
  • space-between：两端对齐，项目之间的间隔都相等。
  • space-around：每个项目两侧的间隔相等。所以，项目之间的间隔比项目与边框的间隔大一倍。

align-items

定义项目在交叉轴上的对齐方式

  • flex-start：交叉轴的起点对齐。
  • flex-end：交叉轴的终点对齐。
  • center：交叉轴的中点对齐。
  • baseline: 项目的第一行文字的基线对齐。
  • stretch（默认值）：如果项目未设置高度或设为auto，将占满整个容器的高度。

align-content

定义多根轴线的对齐方式，如果项目只有一根轴线，该属性不起作用。

  • flex-start：与交叉轴的起点对齐。
  • flex-end：与交叉轴的终点对齐。
  • 与交叉轴的中点对齐。
  • space-between：与交叉轴两端对齐，轴线之间的间隔平均分布。
  • space-around：每根轴线两侧的间隔都相等。所以，轴线之间的间隔比轴线与边框的间隔大一倍。
  • stretch（默认值）：轴线占满整个交叉轴。

子项目的属性

order

定义项目的排列顺序。数字越小，排列越靠前，默认0。

flex-grow

定义项目的放大比例，默认为0，即如果存在剩余空间，也不放大。

如果所有项目的flex-grow属性都为1，则它们将等分剩余空间（如果有的话）。如果一个项目的flex-grow属性为2，其他项目都为1，则前者占据的剩余空间将比其他项多一倍。

flex-shrink

定义项目的缩小比例，默认为1，即如果空间不足，该项目缩小。

如果所有项目的flex-shrink属性都为1，当空间不足时，都将等比例缩小。如果一个项目的flex-shrink属性为0，其他项目都为1，则空间不足时，前者不缩小。

flex-basis

定义在分配多余空间之前，项目占据的主轴空间。浏览器根据这个属性，计算主轴是否有多余空间。默认值为auto，即项目的本来大小。

它可以设为跟width或height属性一样的值（比如350px），则项目将占据固定空间。

flex

是flex-grow、flex-shrink、flex-basis属性的简写，默认值为 0 1 auto。后两个属性可选。

该属性有两个快捷值：auto (1 1 auto) 和 none (0 0 auto)。

align-self

该属性允许单个项目有与其他项目不一样的对齐方式，可覆盖align-items属性。默认值是auto，表示继承父元素的align-items属性，如果没有父元素，则等同于stretch。

该属性可能取6个值，除了auto，其他都与align-items属性完全一致。

CSS知识点总结

发表于 2018-03-05 | 分类于 CSS

盒模型

标准盒子模型：宽度 = 内容宽度(contengt) + border + padding + margin;

低版本IE盒子模型：宽度 = 内容宽度(content + border + padding) + margin;

box-sizing

用来控制元素的盒子模型的解析模式，默认为content-box。

  • content-box：在宽度和高度之外绘制元素的内边框和边框。
  • border-box：为元素设置的任何padding和border都将在已设定的宽度和高度内进行绘制。
  • inherit：规定应从父元素继续box-sizing属性的值。

display属性值

  • inline（默认）–行内元素
  • none–隐藏
  • block–块元素
  • table–表格显示
  • list-item–项目列表
  • inline-block–行内块级元素

块级元素： div、p、h1~h6、form、ul、li、ol、dl、address、hr、menu、table

行内元素： a、span、strong、br、img、input、label、select、textarea、cite

区别：

  • 块级元素会独占一行，其宽度自动填满其父元素宽度。行内元素不会独占一行，相邻的行内元素会排列在同一行里，直到一行排不下，才会换行，其宽度随着元素的内容而变化。
  • 块级元素可以设置width、height属性。行内元素设置宽高无效。
  • 块级元素可以设置margin、padding。行内元素的水平方向的padding、margin可以产生边距效果，但是竖直方向的padding、margin设置无效。
  • 块级元素可以包含块级元素和行内元素，但是行内元素只能包含文本或其他行内元素，不能包含块级元素。

注意： 对于div之间或span之间因标签换行产生的间距，可以通过给父元素设置font-size：0，或者去掉换行，来消除。

display:inline-block产生间隙

  • 有空格的时候会产生间隙。解决办法：移除空格。
  • margin正值的时候产生间隙。解决办法：margin使用负值。
  • 使用font-size的时候产生间隙。解决办法：font-size:0、 letter-space、word-spacing。

position属性值

  • static：（默认）按照正常文档流进行排列。
  • relative：（相对定位）不脱离文档流，参考自身静态位置通过top、bottom、left、right进行定位。(原本的空间还占据着)
  • absolute：（绝对定位）会脱离文档流，参考距离自己最近的一个不为static的父级元素通过top、left、bottom、right进行定位。（不占有原本的空间，后面的元素会顶上去，而且不论元素是行内元素还是块级元素，都会生成一个块级框）
  • fixed：（固定定位）参考可视窗口(浏览器)进行定位。

包含块： 可以理解为一个矩形，其作用为它里面包含的元素提供一个参考，元素的尺寸和位置的计算往往是由该元素所在的包含块决定的。

  • static元素和relative元素：包含块由它最近的块级、单元格或者行内块祖先元素的内容框创建。
  • fixed元素：包含块是当前浏览器窗口。
  • absolute元素：包含块由最近的position不为static的祖先元素创建。如果创建包含块的祖先元素是行内元素，包含块的范围是这个祖先元素中的第一个和最后一个行内块的padding围起来的区域。如果祖先元素不是行内元素，包含块的范围就是这个祖先元素的内边距+width区域。如果没有找到这样的祖先元素，这个绝对定位的元素的包含块为初始块。

position、display、float：
display属性规定元素应该生成的框的类型，position属性规定元素的定位类型；float属性是一种布局方式，定义元素在哪个方向浮动。

position:absolute和float的共同点，可以让元素脱离文档流，并且可以设置宽高。不同点是：float仍会占据位置，absolute会覆盖文档流中的其他元素。

visibility属性值

  • visible：(默认值)元素可见。
  • hidden：元素不可见。
  • collapse：当在表格元素中使用时，此值可删除一行或一列，但是他不会影响表格的布局。被行或列占据的空间会留给其他内容使用。如果此值在其他元素上使用，会呈现为“hidden”。
  • inherit：规定从父元素继承visibility属性值。

collapse浏览器不同表现：

  • chrome中，使用collapse值和使用hidden没有区别。
  • firefox，opera和IE中，使用collapse值和使用display:none没有区别。

dispaly:none和visibility:hidden的区别：

  • display:none，不显示对应的元素，在文档布局中不再分配空间（回流+重绘）。
  • visibility:hidden，隐藏对应元素，在文档布局中仍保留原来的空间（重绘）。

overflow属性值

  • scroll:必会出现滚动条。
  • auto：子元素内容大于父元素时出现滚动条。
  • visible：（默认值）溢出的内容出现在父元素之外。
  • hidden：溢出隐藏。

清除浮动

  • 父级div定义height。
  • 最后一个浮动元素后面加空div，并添加clear:both样式。
  • 包含浮动元素的父元素添加overflow:hidden/auto样式。
  • 父级div加伪类：after。
  • 父级div定义zoom(IE6/7)。

浮动带来的问题：

  • 父元素的高度无法被撑开，影响与父元素同级的元素。
  • 与浮动元素同级的非浮动元素会跟随其后。
  • 若非第一个元素浮动，则该元素之前的元素也需要浮动，否则会影响页面显示的结构。

设置元素浮动后，该元素的display值为block。

BFC(块级格式上下文)

BFC规定了内部块级元素如何布局：

  • 内部的div会在垂直方向上一个接一个放置。
  • div垂直方向的距离由margin决定，属于同一个BFC的两个相邻的div的margin会发生重叠。(取较大的值)
  • 每个元素的margin-left，与包含块的左边相接触。
  • BFC的区域不会与float元素重叠。
  • BFC是页面上的一个隔离的独立容器，容器里面的子元素不会影响到外面的元素。
  • 计算BFC的高度时，浮动元素也会参与计算。

触发BFC的条件：

  • 根元素，即html。
  • float的值不为none(默认值)时。
  • overflow的值不为visible(默认值)时。
  • display的值为inline-block、table-cell、table-caption时。
  • position的值为absolute或fixed时。
  • flex布局(display:flex或inline-flex)。

margin重叠解决方法：

  • 在其中一个元素的外面包裹一层容器，并触发该容器生成一个BFC。那么两个元素便属于不同的BFC，就不会发生margin重叠。
  • 规定css规范，统一margin-top或者margin-bottom.

BFC应用场景：

  • 解决浮动问题。给父元素设置overflow:hidden,触发BFC，所以他内部的元素就不会影响外面的布局。BFC就把浮动的子元素高度当做了自己内部的高度去处理溢出，所以外面看起来是清除了浮动。
  • 解决侵占浮动元素的问题。为非浮动元素建立BFC环境，根据BFC不与float元素重叠的规则，可以解决侵占问题。

IFC(内联格式化上下文)

IFC的line box（线框）高度由其包含行内元素中最高的实际高度计算而来（不受到竖直方向的padding/margin影响)。

每个IFC对外表现为块级元素，与div垂直排列。

使用场景：

  • 水平居中：当一个块要在环境中水平居中时，设置其为inline-block则会在外层产生IFC，通过text-align则可以使其水平居中。
  • 垂直居中：创建一个IFC，用其中一个元素撑开父元素的高度，然后设置其vertical-align:middle，其他行内元素则可以在此父元素下垂直居中。

GFC

GFC(GridLayout Formatting Contexts)直译为”网格布局格式化上下文”，当为一个元素设置display值为grid的时候，此元素将会获得一个独立的渲染区域，我们可以通过在网格容器上定义网格定义行和网格定义列属性，在网格项目上定义网格行和网格列,为每一个网格项目定义位置和空间。

FFC

FFC(Flex Formatting Contexts)直译为”自适应格式化上下文”，display值为flex或者inline-flex的元素将会生成自适应容器。伸缩容器中的每一个子元素都是一个伸缩项目。伸缩项目可以是任意数量的。伸缩容器外和伸缩项目内的一切元素都不受影响。简单地说，Flexbox 定义了伸缩容器内伸缩项目该如何布局。

FOUC

Flash Of Unstyled Content 文档样式闪烁

通过@import引入css文件，浏览器DOM加载完到css加载完成中间会有一段时间页面上的内容是没有样式的，这段时间长度跟网速有关，样式加载完毕后，页面重新渲染，引起重绘，重排。

解决方法：

把@import换成link标签，或者在link后面加一个script标签

margin和padding适用场景

margin的使用：

  • 需要在border外侧添加空白。
  • 空白处不需要背景色。
  • 上下相连的两个盒子之间的空白，需要相互抵消时。

padding的使用：

  • 需要在border内测添加空白。
  • 空白处需要背景色。
  • 上下相连的两个盒子的空白，希望为两者之和时。

元素竖直方向百分比

当按百分比设定一个元素的宽度时，它是相对于父容器的宽度计算的，但是，对于一些表示竖向距离的属性，例如 padding-top , padding-bottom , margin-top , margin-bottom 等，当按百分比设定它们时，依据的也是父容器的宽度，而不是高度。

line-height

行高是指一行文字的高度，具体说是两行文字间基线的距离。CSS中起高度作用的是height和line-height，没有定义height属性，最终其表现作用一定是line-height。

单行文本垂直居中：把line-height值设置为height一样大小的值可以实现单行文字的垂直居中，其实也可以把height删除。
多行文本垂直居中：需要设置display属性为inline-block。

css创建三角形

首先，需要把元素宽度、高度设为0，然后其中三个边框设为透明。

常见兼容性问题

  1. 不同浏览器的标签默认的margin和padding不一样。我们可以设置*{margin: 0; padding: 0}
  2. IE6双边距bug：块属性标签float后，又有水平方向的margin情况下，在IE6显示左边的margin可能会变成两倍宽度。我们可以设置 display: inline，将其转化为行内元素，或者改变padding。
  3. 设置较小高度标签（小于10px），在IE6，IE7中高度超出自己设置的高度。我们可以给超出高度的标签设置 overflow: hidden，或者设置行高line-height小于你设置的高度。
  4. Chrome中文界面下默认会将小于12px的文本强制按照12px显示。我们可以加入css属性 -webkit-text-size-adjust: none(高版本不支持);或者设置transform:scale(0.8)
  5. 超链接访问过后hover样式不见了，被点击访问过的超链接样式不再具有hover和active了。我们可以改变css属性排列顺序：L-V-H-A: a：link{},a:visited{},a:hover{},a:active{}.

视差滚动效果

视差滚动（Parallax Scrolling）通过在网页向下滚动的时候，控制背景的移动速度比前景的移动速度慢来创建出令人惊叹的3D效果。

  • CSS3实现。优点：开发时间短、性能和开发效率比较好，缺点是不能兼容到低版本的浏览器。
  • jQuery实现。通过控制不同层滚动速度，计算每一层的时间，控制滚动效果。优点：能兼容到各个版本的，效果可控性好
    缺点：开发起来对制作者要求高。
  • 插件实现方式
    例如：parallax-scrolling，兼容性十分好。

浏览器怎样解析css选择器

CSS选择器的解析是从右向左解析的。若从左向右的匹配，发现不符合规则，需要进行回溯，会损失很多性能。

若从右向左匹配，先找到所有的最右节点，对于每一个节点，向上寻找其父节点直到找到根元素或满足条件的匹配规则，则结束这个分支的遍历。

两种匹配规则的性能差别很大，是因为从右向左的匹配在第一步就筛选掉了大量的不符合条件的最右节点（叶子节点），而从左向右的匹配规则的性能都浪费在了失败的查找上面。

而在 CSS 解析完毕后，需要将解析的结果与 DOM Tree 的内容一起进行分析建立一棵 Render Tree，最终用来进行绘图。

在建立 Render Tree 时（WebKit 中的「Attachment」过程），浏览器就要为每个 DOM Tree 中的元素根据 CSS 的解析结果（Style Rules）来确定生成怎样的 Render Tree。

在网页中应该使用偶数字体

使用偶数字体。偶数字号相对更容易和 web 设计的其他部分构成比例关系。Windows 自带的点阵宋体（中易宋体）从 Vista 开始只提供 12、14、16 px 这三个大小的点阵，而 13、15、17 px时用的是小一号的点。（即每个字占的空间大了 1 px，但点阵没变），于是略显稀疏。

手写动画，最小时间间隔

多数显示器默认频率是60Hz，即1秒刷新60次，所以理论上最小间隔为1/60＊1000ms ＝ 16.7ms。

li与li之间的空白间隔

行框的排列会受到中间空白（回车空格）等的影响，因为空格也属于字符,这些空白也会被应用样式，占据空间，所以会有间隔，把字符大小设为0，就没有空格了。

解决方法：

  • 可以将li代码全部写在一排。
  • 浮动li中float：left。
  • 在ul中用font-size：0（谷歌不支持）；可以使用letter-space：-3px。

css选择器

id选择器(#myid)、类选择器(.myclassname)、标签选择器(div, h1, p)、相邻选择器(h1 + p)、子选择器（ul > li）、后代选择器（li a）、通配符选择器（*）、属性选择器（a[rel=”external”]）、伪类选择器（a:hover, li:nth-child）

优先级： 从高到低

  • ！important
  • 内联样式(标签的style属性)
  • id选择器
  • class选择器、属性选择器、伪类选择器
  • 标签选择器、伪元素选择器
  • 通配符选择器、子选择器、相邻选择器
  • 选择器相同时，后面的会覆盖前面的
  • 继承的样式优先级最低

权重计算：

把选择器的特殊性分为4个等级，每个等级代表一类选择器：

  • 内联样式的权值：1000
  • ID选择器的权值：100
  • Class选择器的权值：10

单例模式

概念： 单例模式提供了一种将代码组织为一个逻辑单元的手段，这个逻辑单元中的代码可以通过单一变量进行访问。

特点：

  • 可以用来划分命名空间，减少全局变量的数量。
  • 可以把代码组织的更为一致，便于阅读和维护。
  • 可以被实例化，且只能实例化一次。

闭包方法的单例模式

惰性单例模式

惰性单例模式即只有需要用到的时候才实例化一个实例。

应用场景： 登录框。利用分支技术把浏览器间的差异封装在运行期间进行设置。（在脚本加载时一次性地确定针对浏览器的代码）

工厂模式

概念： 提供创建对象的接口，意思就是根据调用者的参数，生成相应的对象。

简单工厂模式：

使用一个类，通常为单体，来生产实例。

优点： 能够解决多个相似的问题。
缺点： 识别不了对象的类型。

复杂工厂模式

将其成员对象的实例化推到子类中，子类可以重写父类接口方法以便创建的时候指定自己的对象类型。

父类只对创建过程中的一般性问题进行处理，这些处理会被子类继承，子类之间是相互独立的，具体的业务逻辑会放在子类中进行编写。

思路：先设计一个抽象类，这个类不能被实例化，只能用来派生子类，最后通过对子类的扩展实现工厂方法。

应用场景： 处理大量具有相同属性的小对象；需要依赖具体环境创建不同实例；对象的构建十分复杂时。

优点： 弱化对象见的耦合，防止代码的重复。在一个方法中进行类的实例化，可以消除重复性的代码。重复性的代码可以放到父类中去编写，子类继承于父类的所有成员属性和方法，子类只专注于实现自己的的业务逻辑。

缺点： 当工厂增加到一定程度的时候，提升了代码的复杂度，可读性下降。而且没有解决对象的识别问题。

观察者模式(发布订阅模式)

概念： 定义了对象间的一种一对多的依赖关系，让多个观察者对象同时监听某一个主题对象，当一个对象的状态发生改变时，所有依赖它的对象都得到通知并自动刷新。

流程：

  • 首先确定谁是发布者
  • 然后给发布者添加一个缓存列表，用于存放回调函数来通知订阅者。
  • 接下来就是发布消息。发布者遍历整个缓存列表，依次触发里面存放的订阅者的回调函数。
  • 最后时退订。

优点： 支持简单的广播通信，当对象状态发生改变时，会自动通知已经订阅过的对象。发布者与订阅者耦合性降低，发布者只管发布一条消息，它不用关系这条消息如何被订阅者使用，同时，订阅者只监听发布者的事件名，不关心它如何发布。

缺点： 创建订阅者需要消耗一定的时间和内存。虽然可以弱化对象之间的联系，如果过度使用的话，反而使代码不好理解以及代码不好维护等。

代理模式

概念： 代理模式就是把对一个对象的访问，交给另一个代理对象来操作。代理是一个对象，它可以用来控制对本体对象的访问，它与本地对象实现了同样的接口，代理对象会把所有的调用方法传递给本体对象；代理模式最基本的形式是对访问进行控制，而本体对象则负责执行所分派的那个对象的函数或者类，简单的来讲本体对象注重的执行页面上的代码，代理对象则控制本地对象何时被实例化。

应用场景： 图片的预加载

在图片加载完成之前，给个loading，加载完成后再替换成实体路径。

这样，myImage只做一件事，只负责将实际图片加入到页面中，而loading图片交给代理对象去做。从而降低代码的耦合度。

优点： 代理对象可以代替本体被实例化，并使其可以被远程访问。它还可以把本体对象实例化推迟到真正需要的时候，对于实例化比较费时的本体对象，或者因为尺寸比较大以至于不适用保存在内存中的本体，我们可以推迟实例化该对象。

应用场景： 缓存代理

缓存代理的含义就是对第一次运行时候进行缓存，当再一次运行相同的时候，直接从缓存里面取，这样做的好处是避免重复一次运算功能，如果运算非常复杂的话，对性能很耗费，那么使用缓存对象可以提高性能。

应用场景： 虚拟代理合并http请求

比如在做后端系统中，有表格数据，每一条数据前面有复选框按钮，当点击复选框按钮时候，需要获取该id后需要传递给给服务器发送ajax请求，服务器端需要记录这条数据，去请求，如果我们每当点击一下向服务器发送一个http请求的话，对于服务器来说压力比较大，网络请求比较频繁，但是如果现在该系统的实时数据不是很高的话，我们可以通过一个代理函数收集一段时间内(比如说2-3秒)的所有id，一次性发ajax请求给服务器，相对来说网络请求降低了, 服务器压力减少了。

策略模式

概念： 策略模式是指定义一系列的算法，把他们一个个封装起来，目的就是将算法的使用和算法的实现分离开来。（说白了就是以前要做很多判定的写法，现在把判定里面的内容抽离开来，变成一个个小的个体）。

应用场景： 比如某个“类”中包含大量的条件性语句，比如if…else或者switch。每一个条件分支都会引起该“类”的特定行为以不同的方式做出改变。与其维护一段庞大的条件性语句，不如将每一个行为划分为多个独立的对象。每一个对象被称为一个策略。设置多个这种策略对象，可以改进我们的代码质量，也更好的进行单元测试。

优点： 策略模式利用组合，委托等技术和思想，有效的避免很多if条件语句。策略模式提供了开放-封闭的原则，使代码更容易理解和扩展。策略模式中的代码可以复用。

VueX原理解读

发表于 2018-03-04 | 分类于 Vue

VueX是一个专门为vue服务，用于管理页面数据状态、提供统一数据操作的生态系统。

基本流程：

  • Vue组件：HTML页面上，负责接收用户操作等交互行为，执行dispatch方法触发对应action进行回应。
  • dispatch：操作行为触发方法，是唯一执行action的方法。
  • actions：操作行为处理模块。负责处理vue组件接收到的所有交互行为。包含同步/异步操作，支持多个同名方法，按照注册的顺序依次触发。向后台API请求的操作就在这个模块中进行，包括触发其他action以及提交mutation的操作。该模块提供了Promise的封装，以支持action的链式触发。
  • commit：状态改变提交方法。对mutation进行提交，是唯一能执行mutation的方法。
  • mutations：状态改变操作方法。是vuex修改state的唯一推荐方法，其他修改行为在严格模式下将会报错。该方法只能进行同步操作，且方法名只能全局唯一。操作之中会有一些hook暴露出来，一进行state的监控等。
  • state：页面状态管理容器对象。集中存储vue组件中data对象的零散数据，全局唯一，以进行统一的状态管理。页面显示所需的数据从该对象中进行读取，利用vue的细粒度数据响应机制来进行高效的状态更新。
  • getters：state对象读取方法。也可以认为是store的计算属性，可以对值进行一些过滤，getter的返回值会根据它的依赖被缓存起来，且只有当它的依赖值发生改变时才会被重新计算。

Vue组件接收交互行为，调用dispatch方法触发action相关处理，若页面状态需要修改，则调用commit方法提交mutation修改state，通过getters获取到state新值，重新渲染vue组件，界面随之更新。

mutations用于更改store中的状态，只有它可以操作store，因为这样可以管理日志，便于调式。

actions可以执行异步任务，mutations只能执行同步任务。

mapState、mapGetters辅助函数，可以将store中state和getter映射到局部计算属性。

在for循环所有监听器的时候，返回一个布尔值，来表示是否还有变更。然后再在外层用一个循环检查这个布尔值的状态，当发生变更的时候，再调用digest函数。(这里digest就是下面的digestOnce())

$digest现在至少运行每个监听器一次，如果第一次运行完，有监控值发生变更了，标记为dirty，所有监听器再运行第二次。这会一直运行，直到所有监控的值不再变化，整个局面就稳定下来了。

作用域本质：添加监听器，在digest里运行它们。

特性：

  • $digest里会调用每个getNewValue()，因此，最好关注监听器的数量，还有每个独立的监控函数或者表达式的性能。

虚拟DOM-React

原理： React先将真实的DOM抽象成一个JavaScript对象，也就是虚拟DOM树，通过虚拟DOM构建正真的DOM；当状态发生变化时，重新构造一颗新的对象树，然后用新的树和旧的树进行比较，记录两棵树的差异；然后批量把所有差异应用到真实的DOM树上。

优点： 最终表现在DOM上的修改只是变更的部分，可以保证非常高效的渲染。

缺点： 首次渲染大量DOM时，由于多了一层虚拟DOM的计算，会比innerHTML插入慢。

diff算法： 首先，每一次生成的虚拟DOM树上的各个节点都对应唯一的一个id，当第二次生成了新的DOM树时， 对原来树上的每一个节点对比新树上对应节点，如果不同，就记录下来这个差异。进行不同的差异进行不同的操作。

在实际的代码中，会对新旧两棵树进行一个深度优先的遍历，这样每个节点都会有一个唯一的标记。在深度优先遍历的时候，每遍历到一个节点就把该节点和新的的树进行对比。如果有差异的话就记录到一个对象里面(这里有差异类型)。

上面说的节点的差异指的是什么呢？对 DOM 操作可能会：

  • 替换掉原来的节点。
  • 移动、删除、新增子节点。
  • 修改了节点的属性。
  • 对于文本节点，文本内容可能会改变。

内容优化

  • 减少HTTP请求数，这条策略是最重要最有效的，因为一个完整的请求要经过NDS寻址，与服务器建立连接，发送数据，等待服务器响应，接收数据这样一个消耗时间成本和资源成本的复杂的过程。常见方法：合并多个CSS文件和js文件，利用CSS Sprites整合图像，Inline Images(使用 data：URL scheme在实际的页面嵌入图像数据 )，合理设置HTTP缓存等。
  • 减少DNS查找。
  • 避免重定向。
  • 使用Ajax缓存。
  • 延迟加载组件，预加载组件。
  • 减少DOM元素数量。因为页面中存在大量的DOM元素，会导致JavaScript遍历DOM的效率变慢。
  • 最小化iframe的数量。iframes提供了一个简单的方式把一个网站的内容嵌入到另一个网站中。但其创建速度比其他包括JavaScript和CSS的DOM元素的创建慢了1-2个数量级。
  • 避免404：HTTP请求时间消耗是很大的，因此使用HTTP请求来获得一个没有用处的响应（例如404没有找到页面）是完全没有必要的，它只会降低用户体验而不会有一点好处。

javascript优化

  • 将JavaScript脚本放在页面的底部.
  • 将JavaScript和CSS作为外部文件来引用：在实际应用中使用外部文件可以提高页面速度，因为JavaScript和CSS文件都能在浏览器中产生缓存。
  • 缩小JavaScript和CSS。
  • 删除重复的脚本。
  • 最小化DOM的访问：使用JavaScript访问DOM元素比较慢。
  • 少用全局变量、缓存DOM查找结果，减少IO读取操作。
  • 开发智能的事件处理程序。
  • javascript代码注意：谨慎使用with,避免使用eval Function函数,减少作用域链查找。

CSS优化

  • 将CSS代码放在HTML页面的顶部。
  • 避免使用CSS表达式。
  • 使用link标签来代替@import。
  • 避免使用Filters。

图像优化

  • 优化图片大小。
  • 通过CSS Sprites优化图片。
  • 不要在HTML中使用缩放图片。
  • favicon.ico要小而且可缓存。

  • 减小Cookie大小。
  • 针对Web组件使用域名无关的Cookie。

  • 使用内容分发网络（CDN）：把网站内容分散到多个、处于不同地域位置的服务器上可以加快下载速度。
  • GZIP压缩。
  • 设置ETag：ETags（Entity tags，实体标签）是web服务器和浏览器用于判断浏览器缓存中的内容和服务器中的原始内容是否匹配的一种机制。
  • 提前刷新缓冲区。
  • 对Ajax请求使用GET方法。
  • 避免空的图像src。

  • 适当使用touch事件代替click事件。

页面从输入URL到完全呈现的过程

发表于 2018-03-01 | 分类于 计算机网络

简单来讲：

浏览器把请求的url交给DNS域名解析，找到真实的IP地址，向服务器发起请求；服务器交给后台处理完成后返回数据，浏览器接收文件（html、js、css、图像等）；浏览器对加载到的资源进行语法解析，建立相应的内部数据结构（如HTML的DOM）；载入解析到的资源文件，渲染页面，完成。

详细过程：

  1. 浏览器接收URL，解析协议、域名、端口号，如果是http协议，默认端口是80，如果是https协议，默认端口是443。
  2. 浏览器将URL与缓存进行对比，如果请求页面在缓存中，且没有过期，则直接返回该页面。
  3. 如果缓存中没有，远程DNS服务器上启动一个NDS查询， DNS域名解析获得请求对应的IP地址。
  4. 浏览器与IP地址所在的服务器通过TCP三次握手协商来建立一个TCP/IP连接。该握手过程是：浏览器发送一个SYN报文，服务器接受报文后回复SYN+ACK报文，最后浏览器接收到ACK报文后也向服务器发送ACK报文。
  5. 一旦TCP/IP连接建立，浏览器会通过该连接向远程服务器发送HTTP请求。数据会经过应用层、传输层、网络层、链路层，传输到下一个目的地。应用层决定了向用户提供应用服务时的通信（HTTP协议、DNS服务）。传输层对上层应用层，提供处于网络连接中的两台计算机之间的数据传输（TCP协议、UDP协议）。网络层用来处理网络上流动的数据包，数据包是网络传输的最小数据单位，该层规定提供怎样的路径达到对方的计算机，并把数据传输给对方。链路层处理连接网络的硬件部分（网卡、光纤）。这一步骤还涉及了发送给服务器的请求报文（请求方法、请求url、协议版本，首部字段、内容实体）。
  6. 服务器收到请求，找到对应的资源并使用HTTP响应返回该资源。响应报文包括协议版本、状态码、解决状态码的原因短语、响应首部字段、实体主体。
  7. 浏览器接到响应报文后，会先检查头部的状态码，针对不同状态码做出不同的反应。
  8. 通过响应首部字段cache-control判断是否可以缓存，如果可以缓存的，这个响应会被存储起来。
  9. 然后，浏览器开始解析响应。
  10. 浏览器会解析HTML生成DOM树；其次会根据CSS生成CSS规则树；然后将DOM和CSS规则树整合形成渲染树；当遇到script标签时，就向服务器去请求下载，如果是纯script标签，这时会阻塞渲染。如果script标签的有defer和asyc属性，表示异步执行，浏览器会把加载和渲染后续元素与加载script脚本并行执行，不会造成渲染阻塞）。
  11. 渲染页面（边解析边渲染）。
  12. 关闭TCP连接或继续保持连接，如果是响应头部有connection：keep-live的话就继续保持连接，如果没有就断开连接。
  13. 四次挥手过程：客户端发送一个FIN报文，服务器响应一个ACK报文，这时服务器还可以继续发送数据，等服务器发送完毕，就给客户端发送一个FIN报文，然后客户端给服务器发送一个ACK报文。连接断开。

defer和asyc的差别：defer是按照加载顺序执行脚本的，asyc是乱序的，一加载完就立刻执行。

window.onload是在页面所有资源(js/css/img)加载完成以后才触发，DOMContentLoaded的触发不需要等待图片资源的加载，也就是说，DOMContentLoaded事件在window.onload事件之前触发。

页面渲染过程

浏览器拿到html文档，解析html代码，自上而下构建DOM树，遇到css/js就去下载这个文件，等到css下载完毕，解析css构建css规则树，然后整合css规则树和DOM树构建渲染树。如果js是同步的，就会阻塞页面渲染，当js加载完毕，浏览器才会向下继续解析。如果是异步的，那么加载和解析并行执行。全部解析完毕，就将渲染树渲染到页面上。

Doctype的作用

什么是<!DOCTYPE html>？

<!DOCTYPE html>声明位于文档中的最前面，处于html标签之前。

它是HTML5标准网页声明，全称为Document Type-Hyper-Text Mark-up Language，意思是文档种类为超文本标记性语言或超文本链接标示语言。现在是这种简洁形式，支持html5标准的主流浏览器都认识这个声明。

此标签可以告知浏览器文档使用哪种HTML或XHTML规范。

HTML 4.01 与 HTML5之间的差异

在 HTML 4.01 中，<!DOCTYPE> 声明需引用 DTD （文档类型声明），因为 HTML 4.01 是基于 SGML （Standard Generalized Markup Language 标准通用标记语言）。DTD 指定了标记语言的规则，确保了浏览器能够正确的渲染内容。

HTML5 不是基于 SGML，因此不要求引用 DTD。

HTML 4.01 规定了三种不同的 <!DOCTYPE> 声明，分别是：Strict、Transitional 和 Frameset。

HTML5 中仅规定了一种：<!DOCTYPE html>

作用：声明文档的解析类型，避免浏览器的怪异模式。

document.compatMode：

  • BackCompat：怪异模式，浏览器使用自己的怪异模式解析渲染页面。
  • CSS1Compat：标准模式，浏览器使用W3C的标准解析渲染页面。

这个属性会被浏览器识别并使用，但是如果页面没有声明Doctype，那么compatMode默认就是BackCompat，就会造成在不同浏览器呈现不同的样式。

XHTML 和HTML

  • XHTML元素必须被正确嵌套；
  • XHTML元素必须被关闭，空标签也必须关闭</\br>；
  • XHTML标签名必须是小写字母；
  • XHTML文档要求必须拥有一个根元素；
  • XHTML文档要求给所有属性赋一个值；
  • XHTML要求所有的属性必须用双引号括起来；
  • XHTML文档需要把所有<、>、&等特殊符号用编码表示；
  • XHTML文档不要在注释中使用“–”；
  • XHTML图片必须有说明文字；
  • XHTML文档中用id属性代替name属性。

XML和JSON

  • 数据体积方面。JSON相对于XML来讲，数据的体积小，传递的速度更快些。
  • 数据交互方面。JSON与JavaScript的交互更加方便，更容易解析处理，更好的数据交互。
  • 数据描述方面。JSON对数据的描述性比XML较差。
  • 传输速度方面。JSON的速度要远远快于XML。

Cookie、LocalStorage、SessionStorage

发表于 2018-02-28 | 分类于 JavaScript ， HTML5

可以通过cookie、localstorage实现多个标签页之间的通信

Cookie

  • 产生由来：作为http规范的一部分而存在，作用是与服务器进行交互。
  • 存储大小：单个域名存储量比较小（各浏览器不同，大致4k）。
  • 有效时期：在设置的cookie过期时间内一直有效。
  • 作用域：在所有同源窗口中都是共享的。
  • 数据传递：会随着请求发送到服务端，在浏览器和服务器之间来回传递。

LocalStorage

  • 产生由来：HTML5引入的，作用是在本地存储用户的浏览数据。
  • 存储大小：单个域名存储量比较大（推荐5MB，各浏览器不同）。
  • 有效时期：永久存储，窗口或浏览器关闭也一直保存。（除非手动删除）
  • 作用域：在所有同源窗口中都是共享的。
  • 数据传递：不会自动把数据发给服务器，仅在本地保存。

SessionStorage

  • 产生由来：HTML5引入的，作用是在本地存储用户的浏览数据。
  • 存储大小：单个域名存储量比较大（推荐5MB，各浏览器不同）
  • 有效时期：只在一个session内有效，当用户关闭浏览器后，数据会被删除。
  • 作用域：只在同一个浏览器的同源窗口共享。
  • 数据传递：不会自动把数据发给服务器，仅在本地保存。

cookie和session

  • cookie数据存在客户端的浏览器上，session数据放在服务器上；
  • session保存的是对象，cookie保存的是字符串；
  • session不能区分路径，cookie可以设置路径；
  • cookie不是很安全，别人可以分析存放在本地的cookie并进行cookie欺骗；
  • session会在一定时间内保存在服务器上，当访问增多，会比较占用服务器资源，影响性能。
  • 单个cookie保存的数据不能超过4kb，很多浏览器都限制一个站点最多保存20个cookie；
  • session需要借助cookie 实现。将sessionId存放在cookie里。

所以将登录等重要信息存放在session中，其他信息如果要保留，可以放在cookie中。

cookie利弊

缺点：

  • Cookie数量和长度的限制。每个domain最多只能有20条cookie，每个cookie长度不能超过4KB，否则会被截掉.
  • 安全性问题。如果cookie被人拦截了，那人就可以取得所有的session信息。即使加密也与事无补，因为拦截者并不需要知道cookie的意义，他只要原样转发cookie就可以达到目的了。
  • 有些状态不可能保存在客户端。例如，为了防止重复提交表单，我们需要在服务器端保存一个计数器。如果我们把这个计数器保存在客户端，那么它起不到任何作用。

优点：极高的扩展性和可用性

  • 通过良好的编程，控制保存在cookie中的session对象的大小。
  • 通过加密和安全传输技术（SSL），减少cookie被破解的可能性。
  • 只在cookie中存放不敏感数据，即使被盗也不会有重大损失。
  • 控制cookie的生命期，使之不会永远有效。偷盗者很可能拿到一个过期的cookie。

IE 提供了一种存储可以持久化用户数据，叫做userdata，从IE5.0就开始支持。每个数据最多128K，每个域名下最多1M。这个持久化数据放在缓存中，如果缓存没有清理，那么会一直存在。

IE和Opera 会清理近期最少使用的cookie，Firefox会随机清理cookie。

TCP三次握手和四次挥手

发表于 2018-02-27 | 分类于 计算机网络

建立TCP需要三次握手才能建立，而断开连接则需要四次握手。整个过程如下图所示：

连接过程：首先Client端发送连接请求报文，Server端接受连接后回复ACK报文，并为这次连接分配资源。Client端接收到ACK报文后也向Server端发送ACK报文，并分配资源，这样TCP连接就建立了。

中断连接的可以是client端，也可以是server端。

断开连接：client端发起中断连接请求，即FIN报文，告诉server端：“我没有数据要发给你了”。server端接到FIN报文后，会先给client端发送一个ACK报文，确认序号为收到序号+1，告诉client端：“我接到你的请求了”。这个时候server端还可以继续发送数据到client端，当server端数据发送完毕，就会向client端发送FIN报文，告诉client端：“我这边数据发送完毕了，准备好关闭连接了”。client端收到FIN报文后，再向server端发送一个ACK报文，确认序号为收到序号+1，告诉server端：“我知道了，你关闭吧”。server端收到ACK报文后，就断开server连接了。这时client端会等待2MSL，如果没有收到回复，就会关闭client连接。到此，TCP连接关闭。

“三次握手”的第二次握手发送SYN+ACK回应第一次握手的SYN，但是“四次挥手”的第二次挥手只能发送ACK回应第一次挥手的FIN，因此，此时server可能还有数据要传输给client，所以server传输完数据后才能发起第三次挥手，等待client的第四次挥手ACK.

按道理，四个报文都发送完毕，就可以直接进入close状态了，但是网络可能出现延迟等现象，造成最后一个ACK丢失。所以client端最后等待的2MSL（2倍的报文最大生存时间）时间就是用来重发可能丢失的ACK报文。

TCP和UDP的区别

TCP

  • 面向连接的。发送数据前需要通过三次握手建立连接；
  • 提供可靠服务。传送的数据时，有确认、窗口、重传、拥塞控制机制，保证数据的有序、无差错、无重复到达。在数据传完后，会断开连接来节约系统资源。
  • 效率低。就是因为发送数据前需要建立连接，所以消耗时间，效率低。
  • TCP面向字节流的。TCP把应用程序看成是一连串的无结构的字节流。TCP有一个缓冲，当应用程序传送的数据块太长，TCP就可以把它划分短一些再传送。如果应用程序一次只发送一个字节，TCP也可以等待积累有足够多的字节后再构成报文段发送出去。
  • 点到点。一条TCP连接只能是点到点的。
  • 应用场景：对网络通讯质量要求高时，比如HTTP、HTTPS、FTP等文件传输的协议，SMTP等邮件传输的协议。

UDP

  • 无连接的。UDP不需要建立连接，可以直接发送数据。
  • 不可靠服务。因为没有TCP那些可靠的机制，在数据传输时，如果网络质量不好，就会很容易丢包。
  • 效率高。不需要建立连接，所以传递数据非常快。
  • 面向报文的。UDP对应用层交下来的报文，既不合并，也不拆分，而是保留这些报文的边界。这也就是说，应用层交给UDP多长的报文，UDP就照样发送，即一次发送一个报文。
  • 多对多。UDP支持一对一、一对多、多对多的交互通信。
  • 应用场景：对网络通信速度要求比较高时，比如QQ语言、视频等。
