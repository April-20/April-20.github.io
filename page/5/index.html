页面从输入URL到完全呈现的过程

发表于 2018-03-01 | 分类于 计算机网络

简单来讲：

浏览器把请求的url交给DNS域名解析，找到真实的IP地址，向服务器发起请求；服务器交给后台处理完成后返回数据，浏览器接收文件（html、js、css、图像等）；浏览器对加载到的资源进行语法解析，建立相应的内部数据结构（如HTML的DOM）；载入解析到的资源文件，渲染页面，完成。

详细过程：

  1. 浏览器接收URL，解析协议、域名、端口号，如果是http协议，默认端口是80，如果是https协议，默认端口是443。
  2. 浏览器将URL与缓存进行对比，如果请求页面在缓存中，且没有过期，则直接返回该页面。
  3. 如果缓存中没有，远程DNS服务器上启动一个NDS查询， DNS域名解析获得请求对应的IP地址。
  4. 浏览器与IP地址所在的服务器通过TCP三次握手协商来建立一个TCP/IP连接。该握手过程是：浏览器发送一个SYN报文，服务器接受报文后回复SYN+ACK报文，最后浏览器接收到ACK报文后也向服务器发送ACK报文。
  5. 一旦TCP/IP连接建立，浏览器会通过该连接向远程服务器发送HTTP请求。数据会经过应用层、传输层、网络层、链路层，传输到下一个目的地。应用层决定了向用户提供应用服务时的通信（HTTP协议、DNS服务）。传输层对上层应用层，提供处于网络连接中的两台计算机之间的数据传输（TCP协议、UDP协议）。网络层用来处理网络上流动的数据包，数据包是网络传输的最小数据单位，该层规定提供怎样的路径达到对方的计算机，并把数据传输给对方。链路层处理连接网络的硬件部分（网卡、光纤）。这一步骤还涉及了发送给服务器的请求报文（请求方法、请求url、协议版本，首部字段、内容实体）。
  6. 服务器收到请求，找到对应的资源并使用HTTP响应返回该资源。响应报文包括协议版本、状态码、解决状态码的原因短语、响应首部字段、实体主体。
  7. 浏览器接到响应报文后，会先检查头部的状态码，针对不同状态码做出不同的反应。
  8. 通过响应首部字段cache-control判断是否可以缓存，如果可以缓存的，这个响应会被存储起来。
  9. 然后，浏览器开始解析响应。
  10. 浏览器会解析HTML生成DOM树；其次会根据CSS生成CSS规则树；然后将DOM和CSS规则树整合形成渲染树；当遇到script标签时，就向服务器去请求下载，如果是纯script标签，这时会阻塞渲染。如果script标签的有defer和asyc属性，表示异步执行，浏览器会把加载和渲染后续元素与加载script脚本并行执行，不会造成渲染阻塞）。
  11. 渲染页面（边解析边渲染）。
  12. 关闭TCP连接或继续保持连接，如果是响应头部有connection：keep-live的话就继续保持连接，如果没有就断开连接。
  13. 四次挥手过程：客户端发送一个FIN报文，服务器响应一个ACK报文，这时服务器还可以继续发送数据，等服务器发送完毕，就给客户端发送一个FIN报文，然后客户端给服务器发送一个ACK报文。连接断开。

defer和asyc的差别：defer是按照加载顺序执行脚本的，asyc是乱序的，一加载完就立刻执行。

window.onload是在页面所有资源(js/css/img)加载完成以后才触发，DOMContentLoaded的触发不需要等待图片资源的加载，也就是说，DOMContentLoaded事件在window.onload事件之前触发。

页面渲染过程

浏览器拿到html文档，解析html代码，自上而下构建DOM树，遇到css/js就去下载这个文件，等到css下载完毕，解析css构建css规则树，然后整合css规则树和DOM树构建渲染树。如果js是同步的，就会阻塞页面渲染，当js加载完毕，浏览器才会向下继续解析。如果是异步的，那么加载和解析并行执行。全部解析完毕，就将渲染树渲染到页面上。

Doctype的作用

发表于 2018-02-28 | 分类于 HTML5

什么是<!DOCTYPE html>？

<!DOCTYPE html>声明位于文档中的最前面，处于html标签之前。

它是HTML5标准网页声明，全称为Document Type-Hyper-Text Mark-up Language，意思是文档种类为超文本标记性语言或超文本链接标示语言。现在是这种简洁形式，支持html5标准的主流浏览器都认识这个声明。

此标签可以告知浏览器文档使用哪种HTML或XHTML规范。

HTML 4.01 与 HTML5之间的差异

在 HTML 4.01 中，<!DOCTYPE> 声明需引用 DTD （文档类型声明），因为 HTML 4.01 是基于 SGML （Standard Generalized Markup Language 标准通用标记语言）。DTD 指定了标记语言的规则，确保了浏览器能够正确的渲染内容。

HTML5 不是基于 SGML，因此不要求引用 DTD。

HTML 4.01 规定了三种不同的 <!DOCTYPE> 声明，分别是：Strict、Transitional 和 Frameset。

HTML5 中仅规定了一种：<!DOCTYPE html>

作用：声明文档的解析类型，避免浏览器的怪异模式。

document.compatMode：

  • BackCompat：怪异模式，浏览器使用自己的怪异模式解析渲染页面。
  • CSS1Compat：标准模式，浏览器使用W3C的标准解析渲染页面。

这个属性会被浏览器识别并使用，但是如果页面没有声明Doctype，那么compatMode默认就是BackCompat，就会造成在不同浏览器呈现不同的样式。

XHTML 和HTML

  • XHTML元素必须被正确嵌套；
  • XHTML元素必须被关闭，空标签也必须关闭</\br>；
  • XHTML标签名必须是小写字母；
  • XHTML文档要求必须拥有一个根元素；
  • XHTML文档要求给所有属性赋一个值；
  • XHTML要求所有的属性必须用双引号括起来；
  • XHTML文档需要把所有<、>、&等特殊符号用编码表示；
  • XHTML文档不要在注释中使用“–”；
  • XHTML图片必须有说明文字；
  • XHTML文档中用id属性代替name属性。

XML和JSON

  • 数据体积方面。JSON相对于XML来讲，数据的体积小，传递的速度更快些。
  • 数据交互方面。JSON与JavaScript的交互更加方便，更容易解析处理，更好的数据交互。
  • 数据描述方面。JSON对数据的描述性比XML较差。
  • 传输速度方面。JSON的速度要远远快于XML。

Cookie、LocalStorage、SessionStorage

发表于 2018-02-28 | 分类于 JavaScript ， HTML5

可以通过cookie、localstorage、sessionstorage实现多个标签页之间的通信

Cookie

  • 产生由来：作为http规范的一部分而存在，作用是与服务器进行交互。
  • 存储大小：单个域名存储量比较小（各浏览器不同，大致4k）。
  • 有效时期：在设置的cookie过期时间内一直有效。
  • 作用域：在所有同源窗口中都是共享的。
  • 数据传递：会随着请求发送到服务端，在浏览器和服务器之间来回传递。

LocalStorage

  • 产生由来：HTML5引入的，作用是在本地存储用户的浏览数据。
  • 存储大小：单个域名存储量比较大（推荐5MB，各浏览器不同）。
  • 有效时期：永久存储，窗口或浏览器关闭也一直保存。（除非手动删除）
  • 作用域：在所有同源窗口中都是共享的。
  • 数据传递：不会自动把数据发给服务器，仅在本地保存。

SessionStorage

  • 产生由来：HTML5引入的，作用是在本地存储用户的浏览数据。
  • 存储大小：单个域名存储量比较大（推荐5MB，各浏览器不同）
  • 有效时期：只在一个session内有效，当用户关闭浏览器后，数据会被删除。
  • 作用域：只在同一个浏览器的同源窗口共享。
  • 数据传递：不会自动把数据发给服务器，仅在本地保存。

Web Storage API

localStorage和sessionStorage具有相同的API。

  • 保存数据：localStorage.setItem(key,value);
  • 读取数据：localStorage.getItem(key);
  • 删除单个数据：localStorage.remove(key);
  • 删除所有数据：localStorage.clear();
  • 得到某个索引的key：localStorage.key(index);

IE浏览器使用UserData来存储数据。

cookie和session

  • cookie数据存在客户端的浏览器上，session数据放在服务器上；
  • session保存的是对象，cookie保存的是字符串；
  • session不能区分路径，cookie可以设置路径；
  • cookie不是很安全，别人可以分析存放在本地的cookie并进行cookie欺骗；
  • session会在一定时间内保存在服务器上，当访问增多，会比较占用服务器资源，影响性能。
  • 单个cookie保存的数据不能超过4kb，很多浏览器都限制一个站点最多保存20个cookie；
  • session需要借助cookie 实现。将sessionId存放在cookie里。

所以将登录等重要信息存放在session中，其他信息如果要保留，可以放在cookie中。

cookie利弊

缺点：

  • Cookie数量和长度的限制。每个domain最多只能有20条cookie，每个cookie长度不能超过4KB，否则会被截掉.
  • 安全性问题。如果cookie被人拦截了，那人就可以取得所有的session信息。即使加密也与事无补，因为拦截者并不需要知道cookie的意义，他只要原样转发cookie就可以达到目的了。
  • 有些状态不可能保存在客户端。例如，为了防止重复提交表单，我们需要在服务器端保存一个计数器。如果我们把这个计数器保存在客户端，那么它起不到任何作用。

优点：极高的扩展性和可用性

  • 通过良好的编程，控制保存在cookie中的session对象的大小。
  • 通过加密和安全传输技术（SSL），减少cookie被破解的可能性。
  • 只在cookie中存放不敏感数据，即使被盗也不会有重大损失。
  • 控制cookie的生命期，使之不会永远有效。偷盗者很可能拿到一个过期的cookie。

IE 提供了一种存储可以持久化用户数据，叫做userdata，从IE5.0就开始支持。每个数据最多128K，每个域名下最多1M。这个持久化数据放在缓存中，如果缓存没有清理，那么会一直存在。

IE和Opera 会清理近期最少使用的cookie，Firefox会随机清理cookie。

TCP三次握手和四次挥手

发表于 2018-02-27 | 分类于 计算机网络

建立TCP需要三次握手才能建立，而断开连接则需要四次握手。整个过程如下图所示：

连接过程：首先Client端发送连接请求报文，Server端接受连接后回复ACK报文，并为这次连接分配资源。Client端接收到ACK报文后也向Server端发送ACK报文，并分配资源，这样TCP连接就建立了。

中断连接的可以是client端，也可以是server端。

断开连接：client端发起中断连接请求，即FIN报文，告诉server端：“我没有数据要发给你了”。server端接到FIN报文后，会先给client端发送一个ACK报文，确认序号为收到序号+1，告诉client端：“我接到你的请求了”。这个时候server端还可以继续发送数据到client端，当server端数据发送完毕，就会向client端发送FIN报文，告诉client端：“我这边数据发送完毕了，准备好关闭连接了”。client端收到FIN报文后，再向server端发送一个ACK报文，确认序号为收到序号+1，告诉server端：“我知道了，你关闭吧”。server端收到ACK报文后，就断开server连接了。这时client端会等待2MSL，如果没有收到回复，就会关闭client连接。到此，TCP连接关闭。

“三次握手”的第二次握手发送SYN+ACK回应第一次握手的SYN，但是“四次挥手”的第二次挥手只能发送ACK回应第一次挥手的FIN，因此，此时server可能还有数据要传输给client，所以server传输完数据后才能发起第三次挥手，等待client的第四次挥手ACK.

按道理，四个报文都发送完毕，就可以直接进入close状态了，但是网络可能出现延迟等现象，造成最后一个ACK丢失。所以client端最后等待的2MSL（2倍的报文最大生存时间）时间就是用来重发可能丢失的ACK报文。

TCP和UDP的区别

TCP

  • 面向连接的。发送数据前需要通过三次握手建立连接；
  • 提供可靠服务。传送的数据时，有确认、窗口、重传、拥塞控制机制，保证数据的有序、无差错、无重复到达。在数据传完后，会断开连接来节约系统资源。
  • 效率低。就是因为发送数据前需要建立连接，所以消耗时间，效率低。
  • TCP面向字节流的。TCP把应用程序看成是一连串的无结构的字节流。TCP有一个缓冲，当应用程序传送的数据块太长，TCP就可以把它划分短一些再传送。如果应用程序一次只发送一个字节，TCP也可以等待积累有足够多的字节后再构成报文段发送出去。
  • 点到点。一条TCP连接只能是点到点的。
  • 应用场景：对网络通讯质量要求高时，比如HTTP、HTTPS、FTP等文件传输的协议，SMTP等邮件传输的协议。

UDP

  • 无连接的。UDP不需要建立连接，可以直接发送数据。
  • 不可靠服务。因为没有TCP那些可靠的机制，在数据传输时，如果网络质量不好，就会很容易丢包。
  • 效率高。不需要建立连接，所以传递数据非常快。
  • 面向报文的。UDP对应用层交下来的报文，既不合并，也不拆分，而是保留这些报文的边界。这也就是说，应用层交给UDP多长的报文，UDP就照样发送，即一次发送一个报文。
  • 多对多。UDP支持一对一、一对多、多对多的交互通信。
  • 应用场景：对网络通信速度要求比较高时，比如QQ语言、视频等。

WebSocket

发表于 2018-02-26 | 分类于 JavaScript ， HTML5

简介

前后端通信方式：Ajax、WebSocket、CORS

WebSocket是HTML5开始提供的一种在单个 TCP 连接上进行全双工通讯的协议。

特点：

  • 它的最大特点就是，服务器可以主动向客户端推送信息，客户端也可以主动向服务器发送信息，是真正的双向平等对话，属于服务器推送技术的一种。
  • 建立在 TCP 协议之上，服务器端的实现比较容易。
  • 与 HTTP 协议有着良好的兼容性。默认端口也是80和443，并且握手阶段采用 HTTP 协议，因此握手时不容易屏蔽，能通过各种 HTTP 代理服务器。
  • 数据格式比较轻量，性能开销小，通信高效。
  • 可以发送文本，也可以发送二进制数据。
  • 没有同源限制，客户端可以与任意服务器通信。
  • 协议标识符是ws（如果加密，则为wss），服务器网址就是 URL。

HTTP 协议有一个缺陷：通信只能由客户端发起。这种单向请求的特点，注定了如果服务器有连续的状态变化，客户端要获知就非常麻烦。我们只能使用”轮询”：每隔一段时候，就发出一个询问，了解服务器有没有新的信息。最典型的场景就是聊天室。（还有一种长连接：和轮询相似，但是为阻塞模式的轮询，客户端请求新的数据request，服务器会阻塞请求，直到有新数据后才返回response给客户端；然后客户端再重复此过程。）

轮询的效率低，非常浪费资源（因为必须不停连接，或者 HTTP 连接始终打开）

WebSocket 属性

Socket.readyState： 表示连接状态，只读。值从0到4变化：

  • 0：表示连接尚未建立。
  • 1：表示连接已建立，可以进行通信。

WebSocket 方法

Socket.close(): 关闭连接。

WebSocket 事件

Socket.onopen： 用于指定连接成功后的回调函数。

Socket.onmessage： 用于指定收到服务器数据后的回调函数。

服务器数据可能是文本，也可能是二进制数据（blob对象或Arraybuffer对象）。

除了动态判断收到的数据类型，也可以使用binaryType属性，显式指定收到的二进制数据类型。

Socket.onerror： 用于指定报错时的回调函数。

Socket.onclose： 用于指定连接关闭后的回调函数。

跨域资源共享CORS

发表于 2018-02-25 | 分类于 JavaScript

简介

CORS是一个W3C标准，全称是”跨域资源共享”（Cross-origin resource sharing）。

它允许浏览器向跨源服务器，发出XMLHttpRequest请求，从而克服了AJAX只能同源使用的限制。

CORS需要浏览器和服务器同时支持。目前，所有浏览器都支持该功能，IE浏览器不能低于IE10。

整个CORS通信过程，都是浏览器自动完成，不需要用户参与。对于开发者来说，CORS通信与同源的AJAX通信没有差别，代码完全一样。浏览器一旦发现AJAX请求跨源，就会自动添加一些附加的头信息，有时还会多出一次附加的请求，但用户不会有感觉。

因此，实现CORS通信的关键是服务器。只要服务器实现了CORS接口，就可以跨源通信。

两种请求

浏览器将CORS请求分成两类：简单请求和非简单请求。

只要同时满足以下两大条件，就属于简单请求。

（1) 请求方法是以下三种方法之一：

  • HEAD
  • GET
  • POST

（2）HTTP的头信息不超出以下几种字段：

  • Accept
  • Accept-Language
  • Content-Language
  • Last-Event-ID
  • Content-Type：只限于三个值application/x-www-form-urlencoded、multipart/form-data、text/plain

凡是不同时满足上面两个条件，就属于非简单请求。

浏览器对这两种请求的处理，是不一样的。

简单请求

对于简单请求，浏览器直接发出CORS请求。具体来说，就是在头信息之中，自动增加一个Origin字段。

上面的头信息中，Origin字段用来说明，本次请求来自哪个源（协议 + 域名 + 端口）。服务器根据这个值，决定是否同意这次请求。

如果Origin指定的源，不在许可范围内，服务器会返回一个正常的HTTP回应。浏览器发现，这个回应的头信息没有包含Access-Control-Allow-Origin字段，就知道出错了，从而抛出一个错误，被XMLHttpRequest的onerror回调函数捕获。注意，这种错误无法通过状态码识别，因为HTTP回应的状态码有可能是200。

如果Origin指定的域名在许可范围内，服务器返回的响应，会多出几个头信息字段。

  • Access-Control-Allow-Origin：该字段是必须的。它的值要么是请求时Origin字段的值，要么是一个*，表示接受任意域名的请求。
  • Access-Control-Allow-Credentials：该字段可选。它的值是一个布尔值，表示是否允许发送Cookie。默认情况下，Cookie不包括在CORS请求之中。设为true，即表示服务器明确许可，Cookie可以包含在请求中，一起发给服务器。这个值也只能设为true，如果服务器不要浏览器发送Cookie，删除该字段即可。
  • Access-Control-Expose-Headers：该字段可选。CORS请求时，XMLHttpRequest对象的getResponseHeader()方法只能拿到6个基本字段：Cache-Control、Content-Language、Content-Type、Expires、Last-Modified、Pragma。如果想拿到其他字段，就必须在Access-Control-Expose-Headers里面指定。上面的例子指定，getResponseHeader(‘FooBar’)可以返回FooBar字段的值。

上面说到，CORS请求默认不发送Cookie和HTTP认证信息。如果要把Cookie发到服务器，一方面要服务器同意，指定Access-Control-Allow-Credentials字段。另一方面，开发者必须在AJAX请求中打开withCredentials属性。

否则，即使服务器同意发送Cookie，浏览器也不会发送。或者，服务器要求设置Cookie，浏览器也不会处理。

需要注意的是，如果要发送Cookie，Access-Control-Allow-Origin就不能设为星号，必须指定明确的、与请求网页一致的域名。同时，Cookie依然遵循同源政策，只有用服务器域名设置的Cookie才会上传，其他域名的Cookie并不会上传，且（跨源）原网页代码中的document.cookie也无法读取服务器域名下的Cookie。

非简单请求

非简单请求是那种对服务器有特殊要求的请求，比如请求方法是PUT或DELETE，或者Content-Type字段的类型是application/json。

非简单请求的CORS请求，会在正式通信之前，增加一次HTTP查询请求，称为”预检”请求。

浏览器先询问服务器，当前网页所在的域名是否在服务器的许可名单之中，以及可以使用哪些HTTP动词和头信息字段。只有得到肯定答复，浏览器才会发出正式的XMLHttpRequest请求，否则就报错。

“预检”请求用的请求方法是OPTIONS，表示这个请求是用来询问的。头信息里面，关键字段是Origin，表示请求来自哪个源。

除了Origin字段，”预检”请求的头信息包括两个特殊字段。

  • Access-Control-Request-Method：该字段是必须的，用来列出浏览器的CORS请求会用到哪些HTTP方法。
  • Access-Control-Request-Headers：
    该字段是一个逗号分隔的字符串，指定浏览器CORS请求会额外发送的头信息字段，上例是X-Custom-Header。

如果浏览器否定了”预检”请求，会返回一个正常的HTTP回应，但是没有任何CORS相关的头信息字段。这时，浏览器就会认定，服务器不同意预检请求，因此触发一个错误，被XMLHttpRequest对象的onerror回调函数捕获。控制台会打印出如下的报错信息。

浏览器同源政策及跨域处理

发表于 2018-02-24 | 分类于 JavaScript

同源政策

同源政策是浏览器为了保证用户信息的安全，防止恶意的网站窃取数据，规定的一种政策。浏览器会限制从脚本内发起的跨域HTTP请求。

所谓同源是指协议、域名、端口相同。

比如http://www.test.com/demo/index.html，其协议是http://，域名是 www.test.com，端口号是80（默认端口号可以省略），它的同源检测为：

  • http://www.test.com/demo/page.html — 同源
  • https://www.test.com/demo/index.html — 不同源，协议不同
  • http://www.study.test.com/demo/index.html — 不同源，域名不同
  • http://www.test.com:8080/demo/index.html — 不同源，端口号不同

如果非同源，以下三种行为会受到限制：

  1. Cookie、LocalStorage和IndexDB无法读取。
  2. DOM无法获得。
  3. Ajax请求不能发送。

不受同源策略限制的行为：

  • 页面中的链接，重定向以及表单提交是不会受到同源策略限制的。
  • script/img/link/iframe等的src资源引入可以跨域，但是js不能读写加载的内容。

JSONP

JSONP是服务器与客户端跨源通信的常用方法。最大特点就是简单适用，老式浏览器全部支持，服务器改造非常小。

基本思想：网页通过添加一个script元素，向服务器请求JSON数据，这种做法不受同源策略的限制；服务器收到请求后，将数据放在一个指定名字的回调函数里传出来。

请求地址中的callback参数用来指定回调函数的名字，这对于JSONP是必须的。

WebSocket

WebSocket是一种通信协议，使用ws://（非加密）和wss://（加密）作为协议前缀。该协议不实行同源策略，只要服务器支持，就可以通过它进行跨源通信。

CORS

CORS是跨域资源共享（Cross-Origin Resource Sharing）的缩写。它是W3C标准，是跨源AJAX请求的根本解决方法。

相比JSONP只能发送get请求，CORS允许发送任何类型的请求。

CORS需要浏览器和服务器同时支持，目前，所有浏览器都支持该功能（IE浏览器不能低于IE10）,整个CORS通信过程都是浏览器自动完成，不需要用户参与。因此，实现CORS通信的关键是服务器。只要服务器实现了CORS接口，就可以跨域通信。

Ajax详解

发表于 2018-02-23 | 分类于 JavaScript

创建XMLHttpRequest对象

Ajax（Asynchronous Javascript And XML）即异步JavaScript和XML.

Ajax不是一种新的编程语言，而是一种用于创建更好更快以及交互性更强的web应用程序的技术。是一套综合多项技术的浏览器网页开发技术。

原理：通过浏览器的XMLHttpRequest对象来向服务器发送异步请求并接收服务器响应数据。然后用JavaScript来操作DOM，实现动态局部刷新。

XMLHttpRequest是一个浏览器接口，使得Javascript可以进行HTTP(S)通信。

XMLHttpRequest对象，用来创建Ajax引擎对象，所有现代浏览器均支持XMLHttpRequest对象。（IE5和IE6使用ActiveXObject）

POST还是GET?

与post相比，get更简单也更快，并且大部分情况下都能用。然而，在以下情况下，请使用post请求：

  • 无法使用缓存文件时（如更新服务器上的文件或数据库）
  • 向服务器发送大量数据时（get有数据量限制）
  • 发送包含未知字符的用户输入时，post比get更稳定也更可靠。

监听响应

在onreadystatechange事件中，我们规定当服务器已做好被处理的准备时所执行的任务。

每当readyState属性改变时，就会触发onreadystatechange事件。

readyState，存有XMLHttpRequest的状态，从0到4发生变化：

  • 0：请求未初始化
  • 1：服务器连接已建立
  • 2：请求已接收
  • 3：请求处理中
  • 4：请求已完成，且响应已就绪.

status,是返回的当前请求的http状态码:

  • 1XX: 消息
    • 100: 初始化请求已经接收,客户继续发送请求剩余部分.
    • 101: 服务器将遵从客户的请求转换成另外一个协议.
  • 2XX: 成功
    • 200: 一切正常,交易成功.
    • 201: 服务器创建了文档.
    • 202: 已经接收请求,但尚未处理
    • 206：客户端只是请求资源的一部分，服务器只对请求的部分资源执行GET方法，相应报文中通过Content-Range指定范围的资源。(通常大资源请求时)
  • 3xx: 重定向
    • 301: 永久重定向(下回不会再请求服务器).
    • 302: 临时重定向(下回依然会请求服务器).
    • 304: 该资源在上次请求之后没有任何修改(这通常用于浏览器的缓存机制).
  • 4xx: 客户端错误
    • 400: 请求语法错误.
    • 401: 请求授权失败.
    • 403: 资源不可用,没有权限访问资源.
    • 404: 访问资源不存在.
    • 407: 访问的资源需要代理身份验证.
    • 414: 请求的url太长.
  • 5XX: 服务端错误
    • 500: 服务器内部错误.
    • 502: 服务器暂时不可用.
    • 503: 服务器由于维护或负载过重没有应答.
    • 504: 关口过载.
  • 6xx: 自定义错误

当readyState=4且status=200或304时表示响应已就绪。206也是响应成功，当发生大资源的断点请求或带宽截流的时候。（后台可以开启gzip压缩）

服务器响应:

  • responseText: 将响应信息作为字符串返回,只读.
  • responseXML: 将响应信息格式化XML Document对象文档并返回,只读.
  • statusText：服务器返回的状态文本。
